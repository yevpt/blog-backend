# OAuth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 Go 后端实现第三方登录与账号绑定，先覆盖旧 Spring Boot 版本的平台能力，再为微信、Google、X 等新平台保留清晰扩展点。

**Architecture:** OAuth 认证身份只使用 `user`、`social_user`、`social_user_auth` 三张表；`user_social_link` 仅作为用户资料展示链接，不进入本次认证链路。协议层独立为 provider adapter，业务层负责登录、自动建号、绑定、解绑和签发本站 JWT。

**Tech Stack:** Gin、GORM、Redis、zap、`golang.org/x/oauth2`、JWT、bcrypt、swaggo/swag。

## Current Best-Practice Decisions

- 使用 Authorization Code Flow，并在支持的平台默认启用 PKCE：`oauth2.GenerateVerifier()`、`oauth2.S256ChallengeOption()`、`oauth2.VerifierOption()`。
- `state` 存 Redis，一次性消费，内容只保存随机 state 对应的上下文：平台、动作、登录用户、PKCE verifier、前端回跳地址。
- 不使用隐式授权，不把第三方 access token 放入 URL，不把第三方 token 返回前端。
- 每个平台配置独立 client id、secret、redirect URI、scope；redirect URI 在平台后台精确登记。
- 不直接采用 Goth 作为业务入口：Goth 可作为 provider 代码参考，但它的 cookie session/global provider 模型不适合作为本 API 项目的核心边界。
- `golang.org/x/oauth2` 只负责标准 OAuth2 换码和 token；平台 userinfo 差异用项目内 adapter 显式处理。
- Google 使用 OIDC 身份语义；GitHub/Gitee/QQ/微博等 OAuth 平台通过 userinfo 接口取 `uuid/email/nickname/avatar/blog`。
- 绑定必须从本站登录态发起；callback 用 Redis state 里的 `user_id` 判定绑定人，不能信任前端传入 user id。
- 解绑只删除 `social_user_auth` 或软删除关联；`social_user` 作为第三方身份历史记录保留，避免重复创建和审计丢失。
- 同一 `source + uuid` 只能绑定一个本站用户；同一本站用户同一 `source` 默认只允许一个绑定账号。

## API Shape

- `GET /oauth/providers`：返回启用平台列表。
- `GET /oauth/:source/authorize?action=login&redirect_uri=...`：返回第三方授权地址。
- `GET /oauth/:source/authorize?action=bind`：需登录，返回绑定授权地址。
- `GET /oauth/:source/callback?code=...&state=...`：处理登录或绑定 callback。
- `GET /oauth/bindings`：需登录，返回当前用户已绑定平台。
- `DELETE /oauth/bindings/:source`：需登录，解绑指定平台。

Callback 响应建议：

- 后端给 API 客户端返回 JSON：登录成功时复用 `dto.LoginResp`。
- 如前端需要页面跳转，再由 `redirect_uri` 白名单决定是否 302 到前端，并只携带一次性短码，不直接带 JWT。

## Phasing

### Phase 1: Minimum Safe OAuth

- 配置、state、manager、GitHub provider、repository、service、handler。
- 目标：GitHub 登录、绑定、解绑跑通。

### Phase 2: Cover Old Spring Boot Platforms

- Gitee 专用 provider。
- QQ、微博、百度、钉钉、支付宝使用专用 adapter；共享 HTTP/token/helper，不共享脆弱字段配置。
- 对每个平台补真实 userinfo 字段映射测试。

### Phase 3: New Platforms

- Google：使用 OIDC，校验 ID token 或 userinfo，`sub` 作为 UUID。
- 微信：按公众号网页授权、开放平台扫码、移动端登录拆成不同 source，避免混淆。
- X：使用 OAuth 2.0 Authorization Code + PKCE；注意 API 权限和邮箱可用性不稳定。
